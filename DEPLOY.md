# Размещение проекта для демонстрации

Проект состоит из нескольких постоянно работающих сервисов:
**frontend** (Next.js), **backend** (FastAPI), **PostgreSQL**, **Redis**,
**worker** (Celery) и **onec-mock** (эмулятор 1С + Mailpit).

## Можно ли на Vercel?
Только фронтенд. Vercel — serverless-хостинг для Next.js и коротких функций;
он не держит PostgreSQL, Redis, Celery-worker и FastAPI как постоянные процессы.
Поэтому весь проект на Vercel не разворачивается. Ниже — рабочие варианты.

---

## Вариант 1. Cloudflare Tunnel / ngrok — быстрее всего «просто показать»
Ничего не хостим: проект крутится у тебя локально, наружу даём временную ссылку.
Подходит для разовой демонстрации (например, защиты).

1. Запусти локально как обычно:
   - `cd app && docker compose up -d`
   - `cd onec-mock && docker compose up -d`
2. Чтобы фронт и API были на одном адресе (иначе придётся возиться с CORS),
   проще пробросить наружу только фронт, а фронт обращается к API через тот же хост.
   Самый простой путь — поставить перед ними один reverse-proxy (см. Вариант 2),
   либо отдать два тоннеля и прописать адрес API фронту (п. «Правки» ниже).
3. Поднять тоннель:
   - Cloudflare: `cloudflared tunnel --url http://localhost:3000`
   - ngrok: `ngrok http 3000`
4. Получишь публичный `https://...`-адрес — открываешь и показываешь.

Плюс: 5 минут. Минус: работает, пока включён твой компьютер и тоннель.

---

## Вариант 2. Один небольшой VPS + docker compose (рекомендую для «живого» демо)
Весь стек уже описан в `app/docker-compose.yml` — на сервере он запускается так же.

1. Арендуй VPS (2 ГБ RAM достаточно), установи Docker и Docker Compose.
2. Скопируй проект на сервер (git clone / scp).
3. Поставь перед сервисами reverse-proxy с авто-HTTPS — удобнее всего **Caddy**:
   один домен на фронт (`:3000`), путь `/api` — на backend (`:8000`).
4. Внеси правки из раздела «Обязательные правки».
5. Запусти: `docker compose up -d --build` (в `app` и в `onec-mock`).

Плюс: всё работает как локально, один публичный адрес. Минус: нужен платный VPS.

---

## Вариант 3. Vercel (фронт) + Railway/Render (бэкенд)
Именно «разнести», как ты спрашивал.

- **Фронт на Vercel:** подключи репозиторий, Root Directory = `app/frontend`,
  переменная окружения `NEXT_PUBLIC_API_URL = https://<адрес-бэкенда>/api`.
- **Бэкенд на Railway или Render** (они умеют контейнеры + managed PostgreSQL и Redis):
  задеплой `backend` (FastAPI), `worker` (Celery), подключи Postgres и Redis,
  задай переменные `DATABASE_URL`, `DATABASE_URL_SYNC`, `REDIS_URL`, `SECRET_KEY`.
- **onec-mock** задеплой тем же способом (отдельный сервис) или укажи его адрес
  в админ-настройках АИС.

Плюс: «правильный» облачный деплой. Минус: больше всего настройки.

## Обязательные правки перед публичным запуском
Сейчас проект настроен под локальный запуск, для внешнего адреса поменяй:

1. **CORS.** `app/backend/app/main.py` — в `allow_origins` добавь публичный адрес
   фронтенда (сейчас там только `http://localhost:3000`).
2. **Адрес API для фронта.** Переменная `NEXT_PUBLIC_API_URL` = публичный адрес
   backend + `/api` (фронт читает её в `app/frontend/src/lib/api.ts`).
3. **Адреса 1С и почты.** В админ-разделе «Интеграция с 1С» укажи доступные адреса:
   - в одном docker-compose — по именам сервисов (`http://onec-mock:8081`);
   - при раздельном хостинге — публичные URL соответствующих сервисов.
   (`host.docker.internal` работает только на локальной машине.)
4. **Секреты.** Задай реальные `SECRET_KEY` (подпись JWT) и пароль БД через
   переменные окружения, не оставляй значения по умолчанию.
5. **Прод-режим фронта.** На хостинге фронт должен собираться и запускаться в
   продакшене (`next build` + `next start`), а не `next dev`.
