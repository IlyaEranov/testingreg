ВВЕДЕНИЕ 
 
 
В последние годы рынок онлайн-продаж и курьерской доставки товаров 
демонстрирует устойчивый рост. Существенным катализатором стала пандемия 
COVID-19, радикально изменившая бизнес-процессы и ускорившая интеграцию 
средств электронной торговли [9]. Из-за массовых ограничений курьерская 
доставка стала одним из немногих доступных каналов сбыта продукции. Даже 
после отмены карантинных мер тенденция к использованию доставки сохраняется, 
свидетельствуя о формировании новых потребительских привычек [13]. Услуга 
«доставка до двери» стала неотъемлемой частью современной модели 
потребления. Для многих магазинов наличие курьерской доставки стало не просто 
дополнительной опцией, а одним из факторов выживания на рынке. 
 
В условиях роста объема заказов на доставку, компании, работающие в сфере 
e-commerce, сталкиваются с необходимостью оптимизации соответствующих 
бизнес-процессов [3]. Эффективное управление заказами, а также интеграция 
систем складского учета с сервисами доставки, такими как «Яндекс.Доставка», 
становятся ключевыми факторами для повышения операционной эффективности и 
улучшения качества обслуживания клиентов. Автоматизация обмена данных 
между системами, позволяет свести ручное вмешательство в процесс оформления 
документов доставки к минимуму, что способствует сокращению временных 
затрат, снижению возможности возникновения ошибок, увеличению скорости 
оперативного реагирования. 
 
Объектами выпускной квалификационной работы являются система 
складского учета «Бизнес.Ру» и логистический сервис «Яндекс.Доставка». 
 
Предметом выпускной квалификационной работы является автоматизация 
обмена данных между системой складского учета «Бизнес.Ру» и логистическим 
сервисом «Яндекс.Доставка» для авторизации процесса формирования документов 
доставки.

1.4 Предлагаемое решение 
 
Для оптимизации бизнес-процесса оформления заявок на доставку 
предлагается разработка и внедрение программного модуля, реализующего

возникновения ошибок за счет автоматизации операций, ранее выполняемых

2.3 Синхронизация программного модуля с сервисами «Бизнес.Ру» и 
«Яндекс.Доставка» 
 
 
Для синхронизации данных модуля с сервисами принято использовать 
механизм вебхуков / callback-вызовов. Это позволит оперативно актуализировать 
данные без необходимости вмешательства персонала. В случае сервиса 
«Бизнес.Ру» сервер подписывается на сообщения об изменениях, происходящих в 
моделях, содержимое которых требуется размещать (актуализировать) в БД 
интеграции [21]. Для «Яндекс.Доставки» при отправке сервисом запросов на 
создание заявок в поле callback_url указывается URL-адрес регистратора [27]. 
Таким образом, интеграция подписывается на оповещения о изменении статусов.

На рисунке 2.3 представлен процесс циклической обработки вебхуков

перезапуска службы (если используется systemd в качестве управляющей 
оболочки). Перед закрытием обработчик завершает последнего взятого в работу 
сообщения, чтобы предотвратить возможное возникновение ошибки или зависание 
в неопределенном состоянии в очереди. 
Перед началом обработки цепочка сообщений, относящихся к аккаунту, 
указанному в теле webhook-сообщения, закрепляется за конкретным экземпляром 
программы

запуска 
нескольких 
идентичных 
экземпляров). Это необходимо для обеспечения линейного порядка обработки 
событий внутри одного аккаунта. Соблюдение последовательности является 
критичным атрибутом: сообщения должны обрабатываться строго в порядке 
возникновения изменений в сервисе «Бизнес.Ру». Нарушение может привести к 
возникновению нежелательных ошибок – например, если изменение склада будет 
зафиксировано до фиксации события его создания.  
После получения сообщения на обработку из его содержимого (поля data) 
вычисляется контрольная сумма – MD5-хэш, который затем сверяется со 
значением, переданным в поле psw. Несовпадение указывает на возможную 
подделку запроса или его модификацию в процессе передачи. В таком случае 
сообщение помечается в очереди как ошибочное, а в журнале лога появляется 
соответствующая запись для последующего анализа. 
Далее в зависимости от типа модели, указанной в теле вебхука, обработчик 
запускает соответствующую подпрограмму, отвечающую за актуализацию данных 
для конкретного вида записей в сервисе «Бизнес.Ру» (рисунок 2.4). В случае 
возникновения допустимой ошибки, интеграция отправляет уведомление согласно 
настройкам.

2.6 Логика работы эндпоинтов заявок на доставку 
 
 
Методы, отвечающие за изменение настроек и получение предзаполненных 
данных через API, не требуют детального описания на этапе проектирования. Их 
поведение может быть формализовано непосредственно на этапе реализации, 
поскольку их в их функционирование не заложена сложная бизнес-логика. В то же 
время, требуется более строгое проектирование операций, связанных с заявками на 
доставку: создание, синхронизация статуса, отмена. Это обусловлено тем, что 
данные действия требуют одновременного взаимодействия с несколькими API, 
базой данных. Недостаточная проработка на этапе проектирования может повлечь 
за собой в будущем возникновение ошибок при кодировании, нарушение 
программной логики, ограничение гибкости из-за неоднозначной трактовки 
процессов.

статуса программным модулем. Ввиду того, что при проектировании учитывается 
возможность работы с несколькими отдельными отправлениями в рамках одного

идентифицировать. К этим параметрам принадлежат: идентификатор заказа,

В качестве хранилища очереди используется таблица MessageQueue в PostgreSQL,

3.6  Логирование и обработка логов 
 
 
Для записи и обработки логов используется стек ELK. С целью снижения 
нагрузки на основной сервер, основные компоненты анализа размещаются на 
отдельном сервере. Логирование осуществляется с использованием пакета python-
logstash-async версии 4.0.2 [29]. Записи отправляются в Logstash по протоколу UDP 
для последующей обработки и индексации в Elasticsearch. В качестве интерфейса 
просмотра и анализа логов используется Kibana (рисунок 3.14). 
Пакет python-logstash-async представляет собой настраиваемый logger и 
formatter, совместимый со стандартным пакетом logging. Запись и отправка 
выполняются в отдельном потоке асинхронно, что позволяет избежать лишних 
задержек при обработке запросов.

class MessageQueue: 
  def __init__(self, db_name: str, db_user: str, db_password: str, db_host: str, db_port: int, cache_maanger): 
    db_url = f"postgresql+psycopg2://{db_user}:{db_password}@{db_host}:{str(db_port)}/{db_name}" 
    self._engine = create_engine(db_url) 
    self._session_factory = sessionmaker(bind=self._engine, expire_on_commit=False) 
    self._cache = cache_maanger 
 
 def add_task( 
   self, message_type: MessageType, task_data: dict | str, start_execute_time: datetime.datetime | None = None, 
 ) -> QueueMessage: 
 
    session = self._session_factory() 
    try: 
      message = QueueMessage( 
        message_type=message_type, task_data=task_data, start_execute_time=start_execute_time 
      ) 
      session.add(message); session.commit(); session.refresh(message) 
      return message 
    except Exception as e: 
      session.close(); raise e 
    finally: 
      session.close() 
 
 def get_next_message(self, message_type: MessageType | None = None) -> QueueMessage | None: 
    session = self._session_factory() 
    try: 
      excluded_ids = [] 
      if message_type == MessageType.WEBHOOK_BRU: 
 
        excluded_ids = [id_.decode() for id_ in self._cache.smembers("queue_locked_account_app_ids")] 
 
      retry_count = 0 
      while True: 
        query = session.query(QueueMessage) 
        if message_type: 
          query = query.filter(QueueMessage.message_type == message_type) 
 
        query = query.filter( 
          or_( 
            and_(QueueMessage.status == MessageStatus.PENDING, QueueMessage.start_execute_time == None), 
            and_(QueueMessage.status == MessageStatus.PENDING, QueueMessage.start_execute_time <= 
datetime.datetime.now()) 
          ) 
        ) 
        if message_type == MessageType.WEBHOOK_BRU and excluded_ids: 
          query = query.filter(not_(QueueMessage.task_data.op('->>')('app_id').in_(excluded_ids))) 
        message = query.order_by(QueueMessage.created.asc()).with_for_update(skip_locked=True).first() 
        if not message: 
          session.close(); return None 
        if message_type == MessageType.WEBHOOK_BRU: 
          app_id = message.task_data.get("app_id") 
          locked = self._cache.set(f"lock:app:{app_id}", "1", nx=True, ex=5) 
          if not locked: 
            retry_count += 1 
            if retry_count % 3 == 0: 
              excluded_ids = [id_.decode() for id_ in self._cache.smembers("queue_locked_account_app_ids")] 
            continue 
          self._cache.sadd("queue_locked_account_app_ids", app_id) 
 
        message.status = MessageStatus.IN_PROGRESS 
        session.commit(); session.expunge(message); session.close() 
        return message 
 
    except Exception as e: 
      session.close(); raise e 
    finally: 
      session.close()

  def update_message_status( 
    self, message_id: int, status: MessageStatus, retry_after_timeinterval: datetime.timedelta = None 
  ) -> bool | None: 
    session = self._session_factory() 
    try: 
      query = session.query(QueueMessage).filter(QueueMessage.id == message_id) 
      message = query.first() 
      if message: 
        message.status = status 
        message.updated = datetime.datetime.now() 
        if status != MessageStatus.PENDING: 
          self._cache.srem("queue_locked_account_app_ids", message.task_data.get("app_id")) 
          self._cache.delete(f"lock:app:{app_id}") 
        if retry_after_timeinterval: 
          message.start_execute_time = datetime.datetime.now() + retry_after_timeinterval 
        session.commit() 
      else: 
        session.close() 
        return None 
    except Exception as e: 
      session.close() 
      raise e 
    finally: 
      session.close() 
      return True 
 
  def delete_completed_messages(self) -> bool | None: 
    session = self._session_factory() 
    try: 
      deleted = session.query(QueueMessage).filter( 
        QueueMessage.status == MessageStatus.COMPLETED 
      ).delete() 
      session.commit() 
    except Exception as e: 
      session.close() 
      raise e 
    finally: 
      session.close() 
      return True
